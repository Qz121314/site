# Cloudflare Personal Catalog Template

一个基于 **Astro SSR + Cloudflare Workers + D1 + R2** 的通用图片目录与转化站模板。

公共站点面向英文内容与美国市场，强调图片浏览、快速进入详情和 CTA 转化；中文管理后台用于维护站点、分区、筛选、分类、产品、图片、联盟广告和转化资源。

本项目不是购物商城。它不提供价格、库存、SKU、购物车、结账、订单、用户账户、收藏、评分或评论。

```text
图片吸引用户
  ↓
快速浏览分区、分类与产品
  ↓
进入独立产品详情页
  ↓
阅读正文并点击 CTA
  ↓
转化池选择可用链接或短信资源
```

## 1. 当前项目状态

当前版本已经完成主要功能链路，并进入 **稳定性、性能与体验收口阶段**。

已建立的核心能力：

- Astro SSR 公共站点与中文管理后台
- 1–4 个分区的内容管理
- 可选分类筛选器与分类目录
- 产品目录、产品详情、搜索与分页加载
- PC 高信息密度目录工作区
- 手机双列分类与双列产品布局
- R2 图片上传、详情图、目录缩略图和删除队列
- D1 Migration、约束、索引和数据完整性检查
- CTA 转化组与链接、短信资源轮换
- 开放式联盟广告组和多种广告素材
- GA4 与 Meta Pixel 按配置加载
- Canonical、Sitemap、Robots、Open Graph 和 JSON-LD
- GitHub Actions 自动检查、部署、生产 Smoke Test 和 Worker 回滚

项目不使用 React，也不是大型 SPA。公共页面由 Astro SSR 输出 HTML，交互使用少量原生 TypeScript/JavaScript。

## 2. 已确定的开发方向

### 2.1 当前阶段目标

当前阶段不继续扩展产品功能，重点是把现有项目优化到可长期稳定运行和可重复部署的状态。

开发优先级固定为：

1. 修复真实 Bug、错误状态和边界问题
2. 清理残留旧代码、重复逻辑和未使用资源
3. 优化 Workers、D1、R2 和日志用量
4. 优化按钮反馈、页面切换、滚动和返回体验
5. 保持手机、平板和 PC 的响应式稳定
6. 统一 UI 密度、间距、图片比例和视觉层级
7. 优化后台数据录入效率与数据一致性
8. 完善 SEO、可访问性和部署验证
9. 完成回归测试后收口

### 2.2 冻结基线

以下内容已经确定，不应在普通优化中重新设计：

- 手机端当前页面结构
- PC 端高信息密度工作区方向
- 站点 → 分区 → 筛选器 → 分类 → 产品的数据关系
- 产品详情使用独立 URL
- 产品详情页保持纯内容与纯转化，不展示广告
- 统一产品模型，不拆成“线上、线下、文章”三套模板
- 一个仓库对应一个 Worker、一个 D1 数据库和一个 R2 Bucket
- 公共站点英文、管理后台中文
- 项目自身不实现访问、曝光、点击或收益统计

### 2.3 当前阶段禁止扩展

除非出现明确业务需求和单独评审，不增加：

- 价格、库存、SKU、购物车、订单和支付
- 用户注册、登录、收藏、评论和评分
- 推荐、猜你喜欢、热榜、复杂排序和个性化
- 广告权重、预算、竞价、地域定向和用户画像
- 多租户、站点市场、插件系统和主题市场
- React、Vue、Hono、ORM 或新的客户端状态框架
- 为重构而重构的大规模架构替换
- 没有数据证明的缓存层和复杂队列

现阶段不新增安全产品功能，但不能削弱现有管理员登录、会话校验、输入校验和第三方代码隔离。

## 3. 产品定位

这是一个可重复部署的通用商业目录模板，可用于：

- 产品推荐
- 服务目录
- 本地商家目录
- 资源导航
- 内容推荐
- 联盟转化站

通用内容结构：

```text
站点
└── 1–4 个分区
    ├── 0–N 个分类筛选器
    ├── 0–N 个分类
    ├── 0–N 个产品
    ├── 0–N 个联盟广告组
    └── 0–N 个 CTA 转化组
```

有效分类的定义：

```text
分类状态 = published
并且
分类下至少存在一个 published 产品
```

分区是否存在有效分类，会影响手机和平板的入口形式：

- 有有效分类：分区首页先展示筛选器和分类入口
- 无有效分类：分区首页直接展示产品目录
- 产品详情页始终使用独立路由

PC 端会把分区导航、筛选、分类和产品预览组织在同一个高密度工作区中，但不取消分类页和产品详情页 URL。

## 4. 公共站点体验

### 4.1 视觉原则

- 白色主背景，辅助区域使用浅灰和冷灰
- 深灰正文、细边框、克制的单一强调色
- 不使用渐变、霓虹、玻璃拟态、粉色主题或厚重阴影
- 使用系统字体，不下载外部字体
- 图片是主要内容，文字保持简短
- 产品卡片统一为图片在上、标题在下
- 不在产品图片上叠加价格、评分、库存或促销徽章
- Header、导航、广告、分类和产品必须有清晰视觉层级
- 所有交互必须保留键盘焦点样式

样式入口统一为：

```text
src/styles/public-system.css
src/styles/admin-system.css
```

Layout 不直接导入内部样式层。样式加载顺序由自动化检查锁定，避免继续叠加无序覆盖。

### 4.2 响应式基线

| 设备 | 断点 | 当前布局 |
|---|---:|---|
| 手机 | `<768px` | 紧凑 Header、筛选器、双列分类、双列产品、底部分区切换 |
| 平板 | `768–1099px` | 独立平板宽度、三列产品、保持原有分类与详情构图 |
| PC | `1100–1359px` | 左侧分区/筛选导航，右侧分类与三列产品 |
| PC | `1360–1759px` | 左侧导航，右侧分类与四列产品 |
| 宽屏 PC | `≥1760px` | 左侧导航，右侧分类与五列产品 |

手机端和平板端属于冻结基线。PC 端优化不能破坏它们的 DOM、按钮顺序、底部导航和页面跳转方式。

### 4.3 手机端

手机端保持当前结构：

```text
Header
广告区域（有可用广告时才创建）
搜索
分类筛选器
双列分类入口
双列产品目录
底部分区切换
```

规则：

- 分类入口固定双列紧凑布局
- 产品目录固定双列
- 底部分区切换支持 1–4 个分区
- 按钮按实际分区数量自动均分，不写死单个宽度
- 产品详情在当前标签页打开
- 产品详情 CTA 可使用移动端固定操作区

### 4.4 PC 端高信息密度工作区

PC 端从 `1100px` 开始使用当前工作区：

```text
顶部栏：Logo | 搜索 | Privacy | Disclaimer

左侧导航               当前分区内容
┌──────────────┐       ┌──────── 分类 ────────┬────── 产品预览 ──────┐
│ 分区 A       │       │ 分类 A   分类 B      │ 产品 1  产品 2 ...  │
│   筛选 A     │       │ 分类 C   分类 D      │                      │
│   筛选 B     │       ├──────────────────────┼──────────────────────┤
│ 分区 B       │       │ 分类 E   分类 F      │ 产品 1  产品 2 ...  │
│   筛选 A     │       └──────────────────────┴──────────────────────┘
└──────────────┘
```

规则：

- 分区按钮不再放在 PC Logo 栏中
- 左侧导航列出所有已发布分区及其有效筛选器
- 当前分区使用明显选中状态
- 当前分区内点击筛选器时进行页面内平滑定位
- 点击其他分区时进入对应分区 URL
- 分类入口保持小尺寸、双列和高信息密度
- 每个筛选器只读取少量产品预览，避免加载完整站点目录
- 产品数量少时不能把单个卡片拉伸占满整行
- 分类页复用同一 PC 工作区，并保留当前分类状态
- 产品详情在 PC 新标签页打开，并使用 `rel="noopener"`

### 4.5 产品图片

目录产品卡片统一使用：

```css
aspect-ratio: 4 / 5;
object-fit: cover;
object-position: center;
```

因此横图、竖图和正方形图片都会进入相同目录框，不允许出现卡片高度不一致。

产品详情主图和图库使用独立展示舞台，优先完整显示素材，不沿用目录裁切逻辑。

推荐素材：

| 用途 | 推荐尺寸 | 最低建议 | 说明 |
|---|---:|---:|---|
| 产品封面原图 | 1200×1500 | 800×1000 | 4:5，JPG、PNG 或 WebP |
| 产品详情图库 | 1200×1500 | 800×1000 | 同一产品尽量统一比例 |
| 目录缩略图 | 最长边约 480px | — | 系统生成独立 WebP |
| Logo | 600×180 | 300×90 | 透明 PNG 或 WebP |
| Favicon | 512×512 | 192×192 | 正方形 PNG 或 WebP |

产品封面必须有目录缩略图。目录查询不读取原图、正文或完整图库。

## 5. 管理后台

管理后台使用中文界面和左侧固定菜单，不使用顶部主导航。

主要管理内容：

- 站点设置
- 分区
- 分类筛选器
- 分类
- 产品与图库
- CTA 转化组
- 联盟广告组和广告
- 图片资产
- Privacy 和 Disclaimer
- GA4 和 Meta Pixel ID

后台数据录入原则：

- 新增与编辑必须回显数据库真实状态
- 表单只显示当前类型需要的字段
- 切换类型时清理不再使用的旧字段
- 列表、编辑页和前台使用同一状态定义
- 保存后立即生效，不要求重新部署
- 删除存在引用的数据时必须明确提示并遵循数据库约束
- 不为偶发操作增加复杂向导或多步骤流程

## 6. CTA 转化系统

转化系统按分区管理：

```text
分区
└── 转化组
    ├── 链接资源
    └── 短信资源
```

产品可绑定一个转化组。用户点击产品详情 CTA 后，由 `/go/*` 路由从当前启用资源中选择可用目标。

规则：

- 一个转化组可以包含多个链接或短信资源
- 资源可单独启用、停用和排序
- 前台不暴露后台资源结构
- 产品没有可用转化资源时，不显示无效 CTA
- 项目自身不记录 CTA 点击统计

## 7. 联盟广告系统

当前广告系统已经使用开放式广告组模型，不再把广告限制为单一 Hero 图片。

### 7.1 数据关系

```text
分区
└── 任意数量广告组
    ├── 设备：mobile / desktop
    ├── 状态：enabled / disabled
    └── 任意数量广告
```

广告支持：

```text
展示类型
- banner
- vertical
- modal

素材类型
- uploaded_image
- external_media
- embed_code
```

平板暂时归入移动端广告设备类型。

### 7.2 页面范围

允许广告：

- 分区首页
- 分类产品目录
- 无分类分区的产品目录
- 搜索结果

禁止广告：

- 产品详情
- Privacy
- Disclaimer
- 后台
- 404 和错误页

产品详情页必须：

- 不查询广告候选数据
- 不创建广告容器
- 不加载远程广告图片或 GIF
- 不执行联盟代码
- 不触发横幅、竖版或弹窗

### 7.3 加载原则

广告初始状态：

```text
无广告 DOM
无占位框
无空白区域
不加载远程素材
不执行联盟代码
```

当前公共页面在主要内容后延迟加载广告脚本，再请求当前分区与设备的候选集合。

要求：

- 每个页面只请求一次广告候选集合
- D1 使用分区、设备、状态和展示类型索引
- Worker 使用随机下标分配，不使用 `ORDER BY RANDOM()`
- 同页多个展示点优先不重复
- 素材加载成功后才创建广告区域
- 加载失败时不留下空白或破图
- 联盟代码通过隔离 iframe 执行，不直接写入主页面 DOM
- 广告链接使用 `rel="sponsored noopener noreferrer"`
- 广告外层明确显示 `Ad` 或 `Sponsored`

### 7.4 推荐广告尺寸

尺寸是运营建议，不作为唯一允许值。系统按实际或声明比例显示，不强制把横幅裁成竖版。

| 设备与形式 | 首选尺寸 | 可接受尺寸 |
|---|---:|---:|
| 移动端横幅 | 300×100 | 320×100、320×50 |
| 移动端竖版 | 300×600 | 300×250 |
| 移动端弹窗 | 300×250 | 最大约 320×480 |
| PC 横幅 | 970×250 | 970×90、728×90 |
| PC 竖版 | 300×600 | 160×600、300×250 |
| PC 弹窗 | 600×400 | 300×250、300×600 |

GIF 建议控制在约 1.5 MB 以内，不使用高频闪烁或自动声音。

## 8. 数据模型

核心关系：

```text
site_settings
├── channels
│   ├── category_filters
│   ├── categories
│   │   └── category_filter_relations
│   ├── products
│   │   ├── product_images
│   │   └── conversion_groups
│   │       └── conversion_resources
│   └── ad_pools
│       └── advertisements
└── image_assets
    └── image_deletion_queue
```

数据库使用原生 SQL、Prepared Statements、外键、检查约束、触发器和索引，不使用 ORM。

当前 Migration 顺序：

```text
0001_initial.sql
0002_affiliate_ad_system.sql
0003_affiliate_ad_rollback_compatibility.sql
```

维护规则：

- 已经发布的 Migration 不直接改写历史语义
- 数据结构变化必须新增 Migration
- Migration 必须同时通过本地 D1 和远程部署流程
- 产品封面必须存在目录缩略图
- 分类、筛选器、产品和转化组必须属于同一分区
- 不能依赖前端校验代替数据库约束
- 不为旧测试数据保留永久兼容字段

## 9. 主要路由

```text
/                                      跳转到默认分区
/{channel}                             分区首页
/{channel}?category={category}         无筛选组模式的分类产品筛选
/{channel}/category/{category}         独立分类产品目录
/{channel}/product/{product}           产品详情，永远不展示广告
/{channel}/search?q=...                搜索结果
/privacy                               隐私政策
/disclaimer                            免责声明
/robots.txt                            Robots
/sitemap.xml                           动态 Sitemap
/admin                                 后台入口
/admin/login                           后台登录
/api/admin/*                           后台接口
/api/public/*                          公共接口
/api/health                            健康检查
/go/*                                  CTA 转化解析
```

分类和产品详情必须保留独立 URL，用于：

- SEO
- 分享
- Canonical
- 浏览器历史
- 返回状态
- 转化归属

PC 端在新标签页打开产品详情属于交互策略，不改变产品详情路由。

- Canonical URL

- Open Graph
- Twitter Card
- 动态 Sitemap

## 12. Cloudflare 架构

```text
GitHub Repository
  ↓
GitHub Actions
  ├── 安装依赖
  ├── 生成 Cloudflare 类型
  ├── Astro / TypeScript 检查
  ├── Node 自动化测试
  ├── 本地 D1 Migration 验证
  ├── 数据完整性验证
  ├── Astro SSR 构建
  ├── Headless Chrome 响应式检查
  ├── Cloudflare Binding 检查
  ├── 记录 Worker 状态和 D1 Time Travel 书签
  ├── 应用远程 D1 Migration
  ├── Wrangler Deploy
  ├── 生产 Smoke Test
  └── 失败时自动回滚 Worker
        ↓
Cloudflare Worker
  ├── Astro SSR
  ├── D1 Binding
  ├── R2 Binding
  └── Static Assets
```

一个仓库只部署：

```text
1 个 Worker
1 个 D1 数据库
1 个 R2 Bucket
```

前台、后台和 API 均由同一个 Worker 提供。

`wrangler.jsonc` 当前启用：

- D1 Binding：`DB`
- R2 Binding：`MEDIA_BUCKET`
- Static Assets Binding：`ASSETS`
- Workers Logs 采样：1%
- Workers Traces 采样：1%

## 13. Cloudflare 免费额度原则

- 首次产品目录读取 20 条
- Load More 每次读取 20 条
- PC 每个筛选器只读取少量产品预览
- 目录查询不读取正文和完整图库
- 目录使用独立 WebP 缩略图
- 视口外图片懒加载
- 图片直接由 R2 公共域名提供
- 页面不加载大型客户端框架
- 广告候选数据每页只请求一次
- 广告素材真正需要展示时才加载
- 页面初始不预留广告空白
- 不使用 `ORDER BY RANDOM()`
- 不为小规模数据增加 Redis、KV 缓存或独立队列
- 日志和 Trace 采样维持 1%，除故障排查外不提高
- 优化必须优先减少 D1 查询次数、返回字段和重复请求

出现性能问题时，先检查：

1. 是否重复查询站点、分区或广告候选
2. 是否读取了正文、原图或完整图库
3. 是否在手机端加载了只供 PC 使用的大量数据
4. 是否有重复客户端脚本或监听器
5. 是否存在无效 DOM、占位区和隐藏大图
6. 是否有不必要的日志和错误堆栈

## 18. 变更规则

所有开发变更应遵循：

1. 从最新 `main` 创建 `agent/*` 分支
2. 一个 PR 只解决一组明确问题
3. 不直接在 `main` 开发功能
4. 先检查现有实现，避免重复组件和重复样式
5. 不修改移动端冻结结构，除非修复明确 Bug
6. 不修改核心数据模型，除非现有结构无法正确表达数据
7. 不增加功能来掩盖交互或布局问题
8. 不以“未来可能使用”为理由保留死代码
9. 更新相关测试、README 和运维说明
10. CI 全绿后再合并
11. 合并后确认生产部署和 Smoke Test

文档修改若被工作流 `paths-ignore` 排除，可以不触发 Worker 部署，但仍需通过 PR 审查和内容核对。

## 19. 收口验收清单

### 公共页面

- 1–4 个分区都能正确展示和切换
- 手机底部分区按钮按数量自动均分
- PC 分区和筛选器位于左侧导航，不出现在 Logo 栏
- 手机分类保持双列紧凑布局
- PC 分类入口尺寸稳定，不出现大型空卡片
- 产品目录图片统一为 4:5
- 产品少时卡片不拉伸
- PC 产品详情新标签页打开
- 手机和平板产品详情当前页打开
- 返回、滚动和筛选状态符合预期
- 无横向溢出、重叠、闪动和大面积空白

### 广告

- 无候选或素材失败时页面不留空白
- 横幅、竖版和弹窗使用正确容器
- 广告不压缩 PC 主内容宽度
- 产品详情、法律页和后台完全不加载广告
- 第三方代码保持 iframe 隔离

### 数据

- 草稿和停用内容不进入公开页面
- 分类、筛选器、产品和转化组分区关系正确
- 产品封面存在目录缩略图
- 编辑表单回显真实状态
- 删除和停用操作不会留下悬空引用

### 性能与部署

- 页面没有重复站点壳、广告和产品查询
- 目录不读取正文、原图和完整图库
- D1 查询使用已有索引
- Workers Logs 和 Traces 保持 1% 采样
- 类型、测试、Migration、构建和浏览器检查通过
- 生产 Smoke Test 通过

完成上述清单后，项目进入维护模式：只处理真实 Bug、兼容性问题、可量化性能问题和必要的 SEO 调整。
